# CLAUDE.md

**절충점(Tradeoff):** 이 지침은 속도보다는 신중함에 우선순위를 둡니다. 사소한 작업의 경우 상황에 맞게 판단하세요.

## 1. 코딩 전 생각하기

**추측하지 마세요. 모호함을 숨기지 마세요. 절충안을 제시하세요.**

구현하기 전에:

* 가정한 사항을 명시적으로 밝히세요. 불확실하다면 물어보세요.
* 해석이 여러 갈래라면 모두 제시하세요. 혼자서 임의로 선택하지 마세요.
* 더 간단한 방법이 있다면 제안하세요. 정당한 근거가 있다면 의견을 피력하세요.
* 불분명한 부분이 있다면 멈추세요. 무엇이 헷갈리는지 짚어내고 질문하세요.

## 2. 단순함 우선

**문제를 해결하는 최소한의 코드만 작성하세요. 추측성 코드는 금물입니다.**

* 요청받지 않은 기능은 추가하지 마세요.
* 한 번만 쓰는 코드에 추상화를 적용하지 마세요.
* 요청받지 않은 "유연성"이나 "설정 가능성"을 고려하지 마세요.
* 발생 불가능한 상황에 대한 에러 처리를 하지 마세요.
* 50줄로 끝낼 수 있는 코드를 200줄로 썼다면, 다시 쓰세요.

스스로에게 물어보세요: "시니어 엔지니어가 보기에 너무 복잡하다고 할까?" 그렇다면 단순화하세요.

## 3. 정밀한 수정 (Surgical Changes)

**꼭 필요한 부분만 건드리세요. 본인이 만든 쓰레기만 치우세요.**

기존 코드를 수정할 때:

* 주변 코드, 주석, 포맷을 "개선"하려 들지 마세요.
* 망가지지 않은 코드를 리팩토링하지 마세요.
* 본인의 스타일과 다르더라도 기존 스타일을 따르세요.
* 관련 없는 안 쓰는 코드(dead code)를 발견하면 언급만 하세요. 직접 삭제하지 마세요.

수정 결과로 인해 고립된 요소가 생길 때:

* **본인의 수정**으로 인해 사용되지 않게 된 import, 변수, 함수는 제거하세요.
* 요청받지 않았다면 기존에 존재하던 안 쓰는 코드는 제거하지 마세요.

**검증 기준:** 변경된 모든 줄은 사용자의 요청과 직접적인 연관이 있어야 합니다.

## 4. 목표 중심 실행

**성공 기준을 정의하세요. 검증될 때까지 반복하세요.**

작업을 검증 가능한 목표로 변환하세요:

* "유효성 검사 추가" → "유효하지 않은 입력에 대한 테스트 작성 후 통과시키기"
* "버그 수정" → "버그를 재현하는 테스트 작성 후 통과시키기"
* "X 리팩토링" → "리팩토링 전후로 테스트 통과 확인"

단계가 많은 작업은 간략한 계획을 세우세요:

```
1. [단계] → 검증: [확인 사항]
2. [단계] → 검증: [확인 사항]
3. [단계] → 검증: [확인 사항]

```

성공 기준이 명확하면 스스로 반복하며 작업을 완성할 수 있습니다. 기준이 모호하면("작동하게 만들기") 계속 다시 물어봐야 합니다.
